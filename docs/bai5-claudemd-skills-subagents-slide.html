CLAUDE.md

Bộ nhớ dự án — chỉ dẫn cố định Claude đọc ở đầu mỗi session

// CLAUDE.MD

File markdown chứa "luật" của dự án

Chính là những gì bạn sẽ phải giải thích lại cho Claude lặp đi lặp lại — viết một lần, dùng mọi session.

Phân biệt với "auto memory"
  • CLAUDE.md — do con người viết để định hướng (luật, quy ước, code style)
  • Auto memory — do Claude tự ghi dựa trên các lần bạn sửa lỗi / sở thích
  • Cả hai đều nạp ở đầu session
// HỆ THỐNG PHÂN CẤP

4 vị trí — nạp từ rộng → hẹp

Phạm viVị tríChia sẻ với
Managed/etc/claude-code/CLAUDE.md (org)Cả tổ chức
User~/.claude/CLAUDE.mdChỉ bạn (mọi dự án)
Project./CLAUDE.mdCả team (qua git)
Local./CLAUDE.local.mdChỉ bạn (gitignore)
Cơ chế: các file được nối (concatenate), KHÔNG ghi đè. File gần nơi chạy Claude đọc sau cùng → ưu tiên logic cao hơn. CLAUDE.local.md KHÔNG bị khai tử.

Đi ngược cây thư mục + import module

  • File ở thư mục con nạp theo nhu cầu (lazy)
  • → Lý tưởng cho monorepo: mỗi team một CLAUDE.md

Giữ ngắn, cụ thể, có cấu trúc

Lệnh hữu ích

Quy tắc vàng

Một CLAUDE.md tốt

# Build & Development
- Dev:   npm run dev  # server ở localhost:3000
- Test:  npm test     # dùng test:e2e cho integration

# Code Conventions
- File: kebab-case · Class: PascalCase · Hàm/biến: camelCase
- Indent 2 spaces, dòng tối đa 100 ký tự, nháy đơn, không semicolon
- React components: src/components/ · API: src/api/

# Before Committing
1. npm test  2. npm run lint  3. commit: type(scope): message

# Important Rules
- KHÔNG commit .env / API key
- Mọi feature mới phải có test · API response luôn xử lý lỗi

# Architecture
- Xem @docs/architecture.md cho thiết kế hệ thống
Vì sao tốt? Cụ thể · có cấu trúc · dùng @import cho chi tiết · < 200 dòng.

Skills

Đóng gói quy trình & kiến thức — nạp theo nhu cầu, tiết kiệm ngữ cảnh

// SKILLS

Biến Claude thành chuyên gia domain

Giải quyết 2 vấn đề
  • Loại bỏ lặp lại — đừng dán lại checklist mỗi lần
  • Chuyên môn hóa — đóng gói instructions, context, best practices, tài liệu
Đóng gói gì?
  • Hướng dẫn từng bước
  • Kiến thức domain
  • Quy trình / best practices
  • API docs, schema, ví dụ
Ví von: Skill giống tài liệu onboarding cho nhân viên mới — có tổ chức, đọc dần khi cần. Cốt lõi: Progressive Disclosure.
// CẤU TRÚC FILE

Mỗi skill là một thư mục + SKILL.md

--- frontmatter (YAML) ---
name: skill-name
description: Skill làm GÌ và
  KHI NÀO Claude nên dùng
--------------------------

# Hướng dẫn chính
Nội dung markdown ở đây...
2 trường BẮT BUỘC
  • name — ≤64 ký tự, chữ thường + gạch nối
  • description — nêu WHAT + WHEN, viết ngôi thứ ba
description là trường QUAN TRỌNG NHẤT — nó quyết định Claude có tự kích hoạt skill không.
// FRONTMATTER · TÙY CHỌN

Điều khiển hành vi skill

TrườngCông dụng
allowed-toolsTool dùng không cần hỏi quyền khi skill bật — vd Bash(git *)
disable-model-invocationtrue → chỉ gọi tay /tên (cho thao tác side-effect: deploy, gửi tin)
user-invocablefalse → ẩn khỏi menu /, chỉ Claude tự nạp (kiến thức nền)
argumentsTham số có tên cho thay thế $tên
context: fork + agentChạy skill trong context cô lập của sub-agent
pathsGlob giới hạn skill chỉ tự kích hoạt với file khớp
Inline: !`git diff HEAD` chạy TRƯỚC khi Claude đọc skill, kết quả thay vào chỗ đó — tiền xử lý, không tốn token để "lấy" dữ liệu.

Skill = thư mục có tài nguyên

Cấp độ

  • .claude/skills/ — dự án (commit git)
  • Plugin — namespace /plugin:skill

  • Chỉ metadata nạp lúc khởi động (~100 token/skill)

Tự động vs thủ công

Tự động

Model quyết định

Claude so khớp task với description của các skill. Khớp → tự nạp.

Chất lượng description quyết định tất cả
Thủ công

Người dùng gõ

Gõ /tên-skill đầu message; truyền tham số:

/fix-issue 123
Điều khiển: disable-model-invocation: true → chỉ gọi tay · user-invocable: false → ẩn khỏi menu, chỉ Claude tự nạp · bỏ trống cả hai = cả hai chiều (mặc định).
// description LÀ TẤT CẢ

WHAT + WHEN + từ khóa người dùng hay nói

❌ Kém — mơ hồ
description: Helps with documents
✅ Tốt — cụ thể, có trigger
description: Extract text and tables from PDF files, fill
  forms, merge documents. Use when working with PDFs or when
  the user mentions PDFs, forms, or document extraction.
Công thức: nêu rõ làm gì (liệt kê output cụ thể) + khi nào dùng (cụm từ người dùng tự nhiên sẽ nói).

// BEST PRACTICES

Viết Skill hiệu quả

  • Ngắn gọn là vua — SKILL.md < 500 dòng
  • description kỹ — what + when + từ khóa
  • SKILL.md như mục lục — chi tiết đẩy sang file riêng
  • Tránh tham chiếu lồng sâu (1 cấp là đủ)
  • Đặt tên dạng gerund: processing-pdfs
  • Luôn dùng dấu / trong path (kể cả Windows)
  • Không thông tin nhạy thời gian
  • Có /skill-creator để tạo/cải thiện
Bảo mật: coi Skill như cài phần mềm — chỉ dùng nguồn tin cậy, audit kỹ SKILL.md + script.

Sub-agents

Trợ lý chuyên biệt với cửa sổ ngữ cảnh RIÊNG

// SUB-AGENTS

Làm việc trong context cô lập

Mỗi sub-agent có riêng
  • System prompt riêng
  • Quyền truy cập tool riêng
  • (Tùy chọn) model, memory, hooks riêng
Tính cô lập
  • Không thấy lịch sử hội thoại của bạn
  • Output dài dòng ở lại context của nó
  • Chỉ trả về tóm tắt → ngữ cảnh chính sạch
Cốt lõi: Claude giao việc (delegate) khi task khớp description; sub-agent làm độc lập rồi trả kết quả.

// FILE ĐỊNH NGHĨA

Markdown + YAML ở .claude/agents/

--- .claude/agents/code-reviewer.md ---
name: code-reviewer
description: Expert code review. Use
  immediately after writing code.
tools: Read, Grep, Glob, Bash
model: sonnet
----------------------------------------

# Code Review Agent
Bạn là senior reviewer...
1. Chạy git diff HEAD
2. Review theo ưu tiên
Trường chính
  • name ✅ · description ✅ (điều khiển delegate)
  • tools — allowlist (bỏ trống = kế thừa hết)
  • disallowedTools — denylist
  • model · permissionMode · memory

Thân markdown = system prompt.

Giao việc khép kín cho context cô lập (research, review nặng)?
   CÓ → Sub-agent

// ANTI-PATTERNS

Những lỗi cần tránh

  1. Cho Claude code khi plan chưa được approve
  2. Đưa UX/UI chi tiết khi không cần — lãng phí token, đóng khung suy nghĩ Claude
  3. Plan dài quá 1 màn hình → context window bị nhiễu, Claude hiểu sai
  4. Skip review user stories vì "thấy ok rồi" → bỏ sót edge cases
  5. 1 prompt khổng lồ chứa nhiều task không liên quan
  6. Không có acceptance criteria → không biết khi nào "xong"
Quy tắc chung: Nếu thấy Claude làm sai hướng, dừng lại — vấn đề thường ở prompt, không phải Claude.

// EXAMPLE

Cùng 1 feature — 2 cách prompt

✕ Prompt tệ
"Tạo trang checkout có form thanh toán, button màu xanh, dùng React, integrate Stripe, validate email với regex"
Chỉ định kỹ thuật + UX + stack. Claude làm theo lệnh, không suy nghĩ. Bị đóng khung từ đầu.
✓ Prompt tốt
"Cần luồng checkout cho e-commerce. Business: support cả guest + logged-in. Thanh toán qua VNPay & MoMo. Giỏ có thể áp voucher (module đã sẵn). Brainstorm 2-3 hướng tiếp cận trước."
Đưa business context + mục tiêu. Mời Claude propose. Bạn review, chọn hướng, đi sâu.
Sự khác biệt: WHAT + WHY thay vì HOW.

// REVIEW

Yêu cầu Claude viết user stories từ plan

  1. Review plan bằng user stories → bạn đóng vai người dùng, dễ phát hiện trường hợp bị bỏ sót.
  2. Review bằng tư duy phản biện:
    • Đọc từng user story, tưởng tượng mình là user — mình sẽ làm gì tiếp?
    • Có trường hợp nào bị sót không?
    • Luôn hỏi "Nếu... thì sao?" để tìm edge cases và xung đột.
// AC

Mỗi user story cần acceptance criteria

User story = WHAT + WHO + WHY.
Acceptance criteria = làm sao biết là "xong" — contract giữa bạn và Claude.

Format Given-When-Then:
GIVEN user đã login & có sản phẩm trong giỏ
WHEN apply mã voucher "SALE10"
THEN discount hiện trên tổng tiền, voucher counter giảm đi 1
AND nếu voucher hết lượt → báo lỗi "Mã đã hết hạn"
Không có AC: Bạn và Claude sẽ hiểu khác nhau về "đã xong". AC = contract, không có nó thì verify cũng vô nghĩa.
// CONTEXT WINDOW

Khi plan quá dài → Claude dễ hiểu sai

Vấn đề: Plan dài → Context Window phình to → Claude mất focus, hiểu sai requirement.
Yêu cầu Claude phân loại tasks thành 3 nhóm:
  1. Có thể implement ngay — task rõ ràng, đủ context → đi vào execute luôn.
  2. Phức tạp, cần plan thêm — Claude ghi vào complex-cases.md: edge cases & decisions cần làm rõ trước khi code.
  3. Cần discuss với team khác — Claude ghi questions.md: câu hỏi gì, ai trả lời, context để họ hiểu.

// CHECKPOINT

Trước khi practice — 6 rule bắt buộc

Vừa đi qua mindset, framework, input & output. Bây giờ là luật chơi trước khi vào tools & thực hành.

6 rule bắt buộc khi plan:
  1. KHÔNG tự động code cho đến khi design được approve.
  2. Luôn dùng plan mode trong giai đoạn này.
  3. Mỗi user story phải có acceptance criteria.
  4. Claude phải dừng và hỏi khi không chắc, không được đoán.
  5. Plan phải được save ra file plan.md — không chỉ tồn tại trong chat.
  6. Không skip plan dù task nhỏ — kỷ luật > tốc độ ngắn hạn.
Design first → Approve → Code. Không bao giờ đảo ngược.

// TOOL

Chia việc với subagents

↗ Hỗ trợ 04 Classify & 05 Execute
  1. 1 plan lớn ≠ 1 prompt khổng lồ. Chia thành các task độc lập, mỗi task có context riêng.
  2. Subagents = agent con xử lý task isolated → không "ô nhiễm" context window chính.
  3. Khi nào delegate:
    • Research độc lập — tổng hợp nhiều file thành 1 câu trả lời
    • Verify / test sau khi implement
    • Task song song không phụ thuộc nhau
Spawn subagent trong Claude Code:
/fork <task description>
Ví dụ: /fork research payment SDK options & tradeoffs
Tip: Sau khi plan xong, hỏi Claude: "Task nào trong plan này có thể /fork?" — Claude sẽ tự nhận ra cấu trúc parallel.
// BEST PRACTICE

Vì sao English là ngôn ngữ tối ưu khi prompt?

Suy nghĩ 30 giây trước khi xem đáp án.

Factor ✓ English Vietnamese / Other
Training data density ~70–80% của hầu hết LLM pretraining corpora < 2% cho tiếng Việt; ngôn ngữ khác cũng thấp
Token efficiency 1 word ≈ 1–1.5 tokens 1 từ tiếng Việt ≈ 2–4 tokens (lãng phí)
Instruction following Benchmark cao nhất trên tất cả models Accuracy giảm với technical instructions
Code alignment Code được viết bằng English → context khớp với code Mismatch giữa ngôn ngữ context & code
RLHF alignment Phần lớn human feedback data là English RLHF coverage sparse cho non-English
→ Always English: prompts · plan files · user stories · code comments.
